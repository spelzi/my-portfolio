# Security Policy

Thank you for taking the time to report a security vulnerability.

The security of this project is taken seriously, and responsible disclosure is greatly appreciated.

---

# Supported Versions

The latest version of this project receives security updates.

| Version        | Supported |
| -------------- | --------- |
| Latest Release | ✅ Yes    |
| Older Versions | ❌ No     |

---

# Reporting a Vulnerability

If you discover a security issue, **please do not create a public GitHub issue**.

Instead, report it privately via one of:

- GitHub's private vulnerability reporting: https://github.com/spelzi/my-portfolio/security/advisories/new
- Email: emmanueluzor1808@gmail.com (subject line: `[SECURITY]`)

Include as much information as possible:

- Description of the vulnerability
- Steps to reproduce
- Potential impact
- Proof of Concept (if available)
- Screenshots or logs (if applicable)

---

# Response Timeline

Security reports are reviewed as quickly as possible.

Typical response timeline:

| Stage                     | Expected Time        |
| ------------------------- | -------------------- |
| Initial acknowledgement   | Within 48 hours      |
| Investigation             | 3–7 business days    |
| Resolution (if confirmed) | As soon as practical |

Complex vulnerabilities may require additional investigation time.

---

# Scope

This policy applies to the entire repository, including but not limited to:

- Frontend application
- Admin dashboard
- Backend API
- Authentication
- Contact form
- File uploads
- Third-party integrations

---

# Out of Scope

The following are generally considered out of scope:

- Issues requiring physical access
- Social engineering attacks
- Denial-of-Service (DoS) testing
- Spam reports
- Vulnerabilities in unsupported versions
- Issues caused solely by outdated browsers

---

# Responsible Disclosure

Please:

- Allow time for investigation before public disclosure.
- Do not access or modify data that does not belong to you.
- Do not intentionally disrupt the service.
- Do not exploit a vulnerability beyond what is necessary to demonstrate its existence.

---

# Best Practices

This project follows several security practices, including:

- Environment variables for sensitive configuration
- Input validation
- Secure API communication
- Dependency updates
- Error handling
- Principle of least privilege
- No hard-coded secrets

---

# Security Updates

Confirmed vulnerabilities will be patched in the next available release.

The project maintainer reserves the right to prioritize fixes based on severity.
